# `init.md`

---

## description: Initialize or update Laravel-specific architecture rules in your architecture_constitution.md.

# Purpose

This command helps you establish or refine Laravel-specific architecture standards within your project's technical constitution (`architecture_constitution.md`).

It ensures that Architecture Guard understands:
* Your **Frontend Stack** (Inertia, Livewire, REST API, or Blade)
* Your **Business Logic Pattern** (Actions, Services, or standard Controllers)
* Your **Validation & Contract Standards** (Form Requests, API Resources, etc.)

---

# Core Principles

## 1. Respect the Stack
Laravel projects vary wildly based on the chosen starter kit (Breeze, Jetstream) and frontend stack (React, Vue, Svelte, Livewire). The rules must match the project's reality.

## 2. Technical vs. Governance
Rules generated by this command belong in `architecture_constitution.md`, not the high-level `constitution.md`.

## 3. Explicit Boundaries
Use Laravel-specific primitives (Controllers, Form Requests, Actions, API Resources) to define architectural boundaries.

# Interview Flow

## Step 1 — Detect Context
1. Check for `architecture_constitution.md`.
2. If it doesn't exist, check for `constitution.md`.
3. If neither exists, warn the user and suggest running `speckit.architecture-guard.init` first to create the base constitution.

## Step 2 — Identify Frontend Stack
Ask the user:
"Which frontend stack are you using for this Laravel project?"
- **Inertia.js (React / Vue / Svelte)**: Modern SPA feel with server-side routing.
- **Livewire**: Reactive UI using PHP and Blade.
- **REST API / SPA**: Headless Laravel with a separate frontend (or Next.js).
- **Blade Monolith**: Traditional server-side rendering.

## Step 3 — Define Logic Ownership
Ask the user:
"How do you prefer to organize business logic?"
- **Actions (Recommended)**: Single-purpose classes (e.g., `CreateUserAction`).
- **Services**: Grouped domain logic (e.g., `UserService`).
- **Domain-Driven**: Strict separation with Domain, Application, and Infrastructure layers.
- **Standard Laravel**: Logic in Controllers or Models (for smaller projects).

## Step 4 — Validation & Contracts
Ask the user:
"What are your standards for validation and API responses?"
- **Strict**: Always use Form Requests and API Resources.
- **Hybrid**: Form Requests for complex logic, inline validation for simple CRUD.
- **Flexible**: Use whatever is most efficient for the task.

# Reference Rules to Generate

Based on the interview, generate a `## Laravel Conventions` section for `architecture_constitution.md`.

## If Inertia.js + Actions
```markdown
## Laravel Conventions (Inertia + Actions)

- **Entry Points**: Controllers must be thin, delegating all business logic to **Actions**.
- **Validation**: Always use **Form Requests** for input validation to keep controllers clean.
- **Contracts**: Use **API Resources** for all Inertia responses to prevent sensitive data leakage (never pass raw Eloquent models).
- **Inertia Data**: Controllers must return `Inertia::render()` and define explicit props.
- **Logic Placement**: Business workflows belong in `app/Actions`. Models must only contain relationships, scopes, and simple accessors.
```

## If Livewire + Services
```markdown
## Laravel Conventions (Livewire + Services)

- **Entry Points**: Livewire components handle UI state; business logic must be delegated to **Services**.
- **Validation**: Use `$this->validate()` for component-specific UI state, but use **Form Requests** or shared rules for business entity validation.
- **Service Layer**: Business logic lives in `app/Services`.
- **Component Scope**: Each Livewire component should focus on a single UI concern (e.g., a specific form or table).
```

## If REST API + Strict
```markdown
## Laravel Conventions (Strict REST API)

- **Entry Points**: API Controllers must follow a strict validate → delegate → respond flow.
- **Validation**: **Form Requests** are mandatory for all POST/PUT/PATCH requests.
- **Response Shape**: Every response must return an **API Resource** or **Resource Collection**.
- **Error Handling**: Use the framework's exception handler; do not use try/catch for flow control in controllers.
```

# Execution Logic

1. **Analyze**: Read existing constitution files to see if Laravel rules already exist.
2. **Propose**: Show the user the proposed additions/updates.
3. **Apply**: Append the new section to `architecture_constitution.md` or update the existing `## Laravel Conventions` section.
4. **Clean up**: If the user has Laravel rules in `constitution.md`, suggest moving them to `architecture_constitution.md` to align with the new split structure.
